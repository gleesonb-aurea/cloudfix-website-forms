## Locked Decisions from CONTEXT.md

**D-01:** Use N8N webhook node's built-in Header Auth option
- Configure header name: `X-Webhook-Secret`
- Configure header value: from `$env.WEBHOOK_SECRET`
- N8N validates automatically before workflow executes

**D-02:** Return HTTP 403 only on validation failure
- No error message body (minimize information leakage)
- No response to form
- No logging of failed attempts

**D-03:** Production testing only
- Test via curl commands against live webhook
- Test cases: valid secret, invalid secret, missing secret
- No staging environment required

**D-04:** No monitoring needed for this phase
- Failed auth attempts handled silently (403)
- N8N's built-in logging sufficient
- No external alerts or IP logging

**D-05:** Rotate on compromise only
- No scheduled rotation
- Document rotation process for future use
- Process: generate new secret, update env var, deploy forms

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → N8N Webhook | Untrusted network, headers can be spoofed but prevents casual scraping |
| N8N Environment Variables | Trusted storage for secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | Webhook endpoint | mitigate | Header Auth validates X-Webhook-Secret against $env.WEBHOOK_SECRET |
| T-01-02 | Tampering | Webhook payload | accept | Out of scope for Phase 1 (HMAC signatures in Phase 2 per WEBHOOK-SECURITY.md) |
| T-01-03 | Repudiation | Form submissions | accept | N8N logs successful workflow executions (failed auths silent per D-04) |
| T-01-04 | Information Disclosure | Error responses | mitigate | Return HTTP 403 only, no message body (per D-02) |
| T-01-05 | Denial of Service | Webhook endpoint | accept | No rate limiting in Phase 1 (future enhancement) |
| T-01-06 | Elevation of Privilege | N/A | N/A | N/A (no auth system, only webhook validation) |

**Security Note:** The WEBHOOK_SECRET is visible in browser dev tools (forms send it via fetch headers). This is acceptable per the threat model - it prevents random scraping and automated bots, but a determined attacker could extract it. Enhanced security (HMAC) is planned for Phase 2 per WEBHOOK-SECURITY.md.
</threat_model>
